# Self-Design Workflow

Bismuth uses its own canvas to design and maintain its UI. This creates a **1-to-1 relationship** between the design artifacts and the running code.

## How It Works

1. **Design on Canvas** — Open the "Bismuth Living Design" canvas to lay out views, components, tokens, and flows
2. **Auto-persist** — Changes save to `.bismuth/design-docs/pages/bismuth-design.json`
3. **Reflect to Code** — Document reflectors extract tokens, components, and layouts back into the codebase
4. **Render with Tokens** — The app's CSS uses the extracted design tokens as its source of truth

## Round-Trip Flow

```
Canvas Editor → Design Document → Reflectors → Code Tokens → Rendered UI
     ↑                                                              |
     └──────────────── Visual inspection ───────────────────────────┘
```

## Canvas Pages

| Page | Contents |
|------|----------|
| App Views | Wireframes of each major view (Welcome, Editor, Canvas, Settings) |
| User Flows | Navigation paths and state transitions |
| Components | Reusable UI components (buttons, inputs, cards, sidebar) |
| Design Tokens | Color palette, spacing scale, typography |
| Architecture | Layer diagram, service boundaries, data flow |

## Using the Self-Design Canvas

1. Switch to **Canvas view** in the sidebar
2. Open or create the "Bismuth Living Design" canvas
3. Use frames to define component boundaries
4. Use text and color swatches to define tokens
5. The design-doc reflectors will extract structured data on save

## Benefits

- **Single source of truth** — design and code stay in sync
- **Visual documentation** — architecture is visible, not just written
- **Iterative** — update the design, see it reflected in the app
- **Portable** — stored as JSON in the vault, version-controlled

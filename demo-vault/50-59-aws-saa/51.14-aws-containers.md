# AWS Containers - ECS, EKS, Fargate

Container orchestration services on [[50-59-aws-saa/51.00-aws-saa-overview|AWS]].

## Docker on AWS

- **ECR** (Elastic Container Registry) — store Docker images (private or public via ECR Public Gallery)
- Docker images are built and pushed to ECR, then pulled by container services

## ECS - Elastic Container Service

AWS's own container orchestration platform.

### Launch Types

**EC2 Launch Type**:

- You provision and maintain [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] instances (infrastructure)
- Each EC2 runs the ECS Agent to register with the ECS cluster
- AWS handles starting/stopping containers

**Fargate Launch Type** (serverless):

- No infrastructure to manage — just create task definitions
- AWS runs containers based on CPU/RAM you specify
- Scaling = increase number of tasks

### ECS Key Concepts

- **Task Definition** — JSON blueprint (Docker image, port, CPU/RAM, env vars, logging)
- **Service** — ensures desired number of tasks running, integrates with [[50-59-aws-saa/51.05-aws-elb|ALB/NLB]]
- **Cluster** — logical grouping of tasks/services

### ECS Integrations

- [[50-59-aws-saa/51.05-aws-elb|ALB]] — most common, dynamic port mapping
- [[50-59-aws-saa/51.05-aws-elb|NLB]] — high throughput / high performance
- [[50-59-aws-saa/51.04-aws-ec2-storage|EFS]] — shared file storage across tasks in any AZ (Fargate + EFS = fully serverless)
- [[50-59-aws-saa/51.08-aws-s3|S3]] — NOT mounted as file system
- [[50-59-aws-saa/51.01-aws-iam|IAM]] Roles:
  - **EC2 Instance Profile** — for ECS Agent (pull images, push logs)
  - **ECS Task Role** — per-task permissions (access S3, DynamoDB, etc.)

### ECS Auto Scaling

- Scale at the **task level** (not EC2): target tracking, step, or scheduled
- Target metrics: CPU utilization, memory utilization, ALB request count per target
- EC2 launch type: use [[50-59-aws-saa/51.06-aws-asg|ASG]] for underlying EC2 scaling, or **ECS Cluster Capacity Provider** (recommended)

## EKS - Elastic Kubernetes Service

Managed Kubernetes on AWS. Use if already using Kubernetes or want Kubernetes API.

- Runs across **multiple AZs** for HA
- Supports EC2 launch type and Fargate
- Node types: **Managed Node Groups** (AWS manages EC2), **Self-Managed Nodes**, **Fargate** (serverless)
- Kubernetes is cloud-agnostic (migrating between cloud providers)

## App Runner

Fully managed service for deploying web apps and APIs at scale.

- Source: container image or source code
- Auto-scaling, load balancing, encryption, [[50-59-aws-saa/51.16-aws-vpc|VPC]] access
- No infrastructure experience needed
- Use for: web apps, APIs, microservices, rapid production deployments

## Flashcards

What is the key difference between ECS and EKS? :: ECS is AWS-proprietary container orchestration; EKS is managed Kubernetes (open-source, cloud-agnostic)

AWS Fargate is ==serverless== — you don't provision or manage EC2 instances for your containers.

ECS Launch Types ::: EC2 Launch Type (you manage EC2 instances) vs Fargate Launch Type (serverless, AWS manages infrastructure) — both run Docker containers

Amazon ECR stores ==Docker images== and can be backed by either S3 or a private/public registry.

### ECS EC2 vs ECS Fargate vs EKS — when to use each? #card #aws #containers

ECS EC2: full control over instances, cost-optimized for steady workloads. Fargate: serverless containers, no instance management, pay per task. EKS: Kubernetes compatibility needed, multi-cloud portability, existing K8s expertise.

ECS tasks need an ==IAM Task Role== to access AWS services like S3 or DynamoDB — each task can have a different role.

ECS with EC2 Launch Type requires running an ==ECS Agent== on each EC2 instance to register it with the cluster.

ECS Service Auto Scaling uses ==AWS Application Auto Scaling== with target tracking, step scaling, or scheduled scaling on CPU, memory, or ALB request count.

ECS Fargate tasks are billed based on ==vCPU and memory== allocated to the task, not by EC2 instance.

EKS supports both ==EC2== and ==Fargate== launch types for running Kubernetes pods.

EKS Node Groups :: Managed groups of EC2 instances that EKS uses as Kubernetes worker nodes — supports managed, self-managed, and Fargate profiles

Amazon ECR images are scanned for ==vulnerabilities== on push and stored backed by ==S3==.

AWS Copilot :: CLI tool to build, release, and operate production-ready containerized apps on ECS and Fargate — simplifies Dockerfile, task definitions, and CI/CD

AWS App Runner :: Fully managed service to deploy web apps and APIs from source code or container images — auto-scaling, load balancing, encryption, no infrastructure experience needed

# AWS CloudFront & Global Accelerator

Content delivery and network acceleration at the [[50-59-aws-saa/51.00-aws-saa-overview#Edge Locations|edge]].

## CloudFront - CDN

Content Delivery Network — caches content at edge locations worldwide.

- 216+ points of presence globally
- DDoS protection via [[50-59-aws-saa/51.15-aws-security#Shield|Shield]] and [[50-59-aws-saa/51.15-aws-security#WAF|WAF]]
- Can expose external HTTPS and talk to internal HTTPS backends

### Origins

- **[[50-59-aws-saa/51.08-aws-s3|S3]] Bucket** — distribute files, enhanced security with Origin Access Control (OAC). Can use as ingress to upload to S3.
- **Custom Origin (HTTP)** — [[50-59-aws-saa/51.05-aws-elb|ALB]], [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]], [[50-59-aws-saa/51.08-aws-s3|S3]] website, any HTTP backend.

### CloudFront vs S3 Cross Region Replication

- **CloudFront** — global edge network, files cached with TTL, great for **static content** accessed globally
- **S3 CRR** — must be set up per region, near real-time updates, great for **dynamic content** needed in few regions

### Caching

- Cache based on: headers, cookies, query string parameters
- Cache lives at each edge location
- **Cache Invalidation** — force refresh by creating invalidation (`/*` for all, `/images/*` for specific)
- Maximize cache hits by separating static (no headers/session caching) and dynamic (cache based on correct headers)

### CloudFront Signed URLs / Signed Cookies

- Distribute paid/restricted content
- **Signed URL** — access to individual files (one URL per file)
- **Signed Cookie** — access to multiple restricted files (one cookie for many files)
- Attach a policy: URL expiration, IP ranges, trusted signers
- Signed URL vs [[50-59-aws-saa/51.08-aws-s3#S3 Pre-Signed URLs|S3 Pre-Signed URLs]]: CloudFront has path-based access to origin; S3 pre-signed issues as the IAM user who signed

### Price Classes

- **All** — all edge locations (best performance)
- **200** — most regions, excludes most expensive
- **100** — only least expensive regions

### Origin Groups

For high availability: primary + secondary origin. Failover on primary failure.

### Field-Level Encryption

Extra security layer: encrypt specific fields at edge using public key. Only application server decrypts.

## Global Accelerator

Uses AWS global network to optimize route to your application. Not a CDN — does NOT cache content.

- Provides **2 static Anycast IPs** that route to edge locations
- Edge locations send traffic over AWS private network to your app
- Health checks + automatic failover (<1 min)
- No client cache issues (IPs don't change)
- Works with: [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]], Elastic IPs, [[50-59-aws-saa/51.05-aws-elb|ALB/NLB]] (public or private)
- [[50-59-aws-saa/51.15-aws-security#Shield|Shield]] DDoS protection

### CloudFront vs Global Accelerator

| Feature      | CloudFront            | Global Accelerator                      |
| ------------ | --------------------- | --------------------------------------- |
| **Purpose**  | Cache content at edge | Optimize network path                   |
| **Protocol** | HTTP/HTTPS            | TCP/UDP (any)                           |
| **IPs**      | Dynamic               | 2 static Anycast                        |
| **Caching**  | Yes                   | No                                      |
| **Best for** | Static content        | Gaming, IoT, VoIP, HTTP with static IPs |

## Flashcards

What is the difference between CloudFront Signed URLs and S3 Pre-Signed URLs? :: CloudFront Signed URLs allow access to any origin via edge caching with IP filtering; S3 Pre-Signed URLs issue requests as the signing IAM user directly to S3

CloudFront caches content at ==edge locations== (216+ globally), while S3 Cross-Region Replication requires setup for ==each target region==.

Global Accelerator ::: Routes traffic through AWS internal network to the nearest optimal endpoint using 2 static Anycast IPs — improves performance for non-HTTP use cases (TCP/UDP, gaming, IoT)

CloudFront Origin Access Control (OAC) replaces ==OAI== to restrict S3 bucket access to CloudFront only.

### CloudFront vs S3 Cross-Region Replication — when to use each? #card #aws #cdn

CloudFront: static content cached at 400+ edge locations worldwide, with TTL. S3 CRR: near-real-time replication to specific regions for dynamic, low-latency reads — must be set up per region.

CloudFront origins can be :: S3 buckets (with OAC for security), ALB, EC2 instances, S3 static websites, or any custom HTTP backend

CloudFront Signed URL :: Provides access to one individual file — attach a policy with URL expiration, allowed IP ranges, and trusted signers

CloudFront Signed Cookie :: Provides access to multiple restricted files with a single cookie — useful when you don't want to change URLs

CloudFront cache can be invalidated by creating an invalidation with paths like ==`/*`== for all files or ==`/images/*`== for specific prefixes.

### What are the three CloudFront Price Classes? #card #aws #cdn

Price Class All: all edge locations (best performance, highest cost). Price Class 200: most regions, excludes the most expensive. Price Class 100: only the least expensive regions.

CloudFront Origin Groups :: Primary + secondary origin for high availability — automatic failover to secondary origin when primary fails

CloudFront Field-Level Encryption :: Encrypt specific sensitive fields at the edge using a public key — only your application server can decrypt with the private key, adding protection beyond HTTPS

Global Accelerator provides ==2 static Anycast IPs== that route to the nearest edge location, then traffic travels over the ==AWS private network==.

Global Accelerator health checks provide automatic failover in less than ==1 minute==.

### CloudFront vs Global Accelerator — when to use each? #card #aws #cdn

CloudFront: caches HTTP/HTTPS content at edge for static assets. Global Accelerator: optimizes network path using AWS backbone for TCP/UDP — ideal for gaming, IoT, VoIP, or any app needing static IPs. Neither caches the same way.

To maximize CloudFront cache hits :: Separate static requests (cache without headers/cookies) from dynamic requests (cache based on specific headers and cookies)

CloudFront can use S3 as an ==ingress== point — meaning you can upload files to S3 through CloudFront.

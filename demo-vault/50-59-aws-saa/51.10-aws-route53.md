# AWS Route 53

Highly available, scalable, fully managed, **authoritative** DNS service. A [[50-59-aws-saa/51.00-aws-saa-overview#Global Services|global service]].

## DNS Basics

- **Domain Registrar** — register domain names (Route 53, GoDaddy, etc.)
- **DNS Records** — A, AAAA, CNAME, NS, Alias, etc.
- **Hosted Zone** — container for DNS records ($0.50/month per hosted zone)
  - **Public** — resolves from internet
  - **Private** — resolves within [[50-59-aws-saa/51.16-aws-vpc|VPC]]

## Record Types

- **A** — hostname to IPv4
- **AAAA** — hostname to IPv6
- **CNAME** — hostname to hostname (can't be at zone apex)
- **NS** — name servers for hosted zone
- **Alias** — hostname to AWS resource (free, native health checks). Works at zone apex. Can't set TTL.

### Alias Targets

[[50-59-aws-saa/51.05-aws-elb|ELB]], [[50-59-aws-saa/51.11-aws-cloudfront|CloudFront]], API Gateway, Elastic Beanstalk, [[50-59-aws-saa/51.08-aws-s3|S3]] Websites, [[50-59-aws-saa/51.16-aws-vpc|VPC]] Interface Endpoints, Global Accelerator, Route 53 records. **Cannot** alias an [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] DNS name.

## Routing Policies

### Simple

Map to one or multiple values. If multiple, client chooses randomly. No health checks.

### Weighted

Control % of requests to each resource. Use for: load balancing, testing new app versions.

### Latency-Based

Route to resource with **lowest latency** for the user. Based on traffic between user and AWS regions.

### Failover

Active-passive. If primary health check fails, route to secondary (DR).

### Geolocation

Route based on **user location** (continent, country, state). Must have a default record.

### Geoproximity

Route based on geographic location with **bias** to shift traffic toward or away from resources. Requires Route 53 Traffic Flow.

### IP-Based

Route based on client IP ranges (CIDR). Use for: ISP-specific routing.

### Multi-Value

Return up to 8 healthy resources. Not a substitute for [[50-59-aws-saa/51.05-aws-elb|ELB]] but gives client-side load balancing with health checks.

## Health Checks

- Monitor endpoints (HTTP, HTTPS, TCP)
- **Calculated health checks** — combine multiple checks with AND/OR/NOT
- Can monitor [[50-59-aws-saa/51.19-aws-monitoring|CloudWatch]] alarms (for private resources)
- 15 global health checkers; endpoint healthy if >18% report healthy

## Route 53 for Disaster Recovery

Route 53 offers personal DNS for [[50-59-aws-saa/51.17-aws-disaster-recovery|disaster recovery]] — migrate DNS between regions.

## Flashcards

What is an Alias record and why use it over CNAME? :: Alias maps hostname to AWS resource, works at zone apex, is free, and has native health checks — CNAME can't be used at zone apex

Route 53 is a ==global== service and an ==authoritative== DNS — you can update DNS records directly.

Route 53 Geoproximity Routing ::: Routes traffic based on geographic location with configurable bias to shift traffic toward or away from resources — requires Traffic Flow

An endpoint is considered healthy if more than ==18%== of global health checkers report it as healthy.

### When do you use each Route 53 routing policy? #card #aws #dns
Simple: single resource. Weighted: A/B testing (split traffic by %). Latency: route to lowest-latency region. Failover: active-passive DR. Geolocation: route by user country/continent. Multi-Value: return up to 8 healthy records.

A hosted zone costs ==$0.50/month==.

CNAME records map hostname to hostname but ==cannot== be used at the zone apex (e.g., example.com).

Alias records ==cannot== target an EC2 DNS name — they work with ELB, CloudFront, API Gateway, S3 websites, VPC endpoints, and Global Accelerator.

Alias records have the TTL set ==automatically by Route 53== — you can't set it yourself.

Route 53 Weighted Routing :: Control the percentage of requests going to each resource — useful for load balancing between regions or testing new app versions

Route 53 Latency-Based Routing :: Routes users to the AWS resource with the lowest network latency based on traffic measurements between user and AWS regions

Route 53 Failover Routing :: Active-passive setup — if the primary health check fails, traffic automatically routes to the secondary (disaster recovery)

Route 53 Geolocation Routing :: Routes based on user's physical location (continent, country, state) — must define a default record for users with no location match

Route 53 Multi-Value Routing :: Returns up to 8 healthy records per query with associated health checks — provides client-side load balancing but is NOT a replacement for ELB

Route 53 IP-Based Routing :: Routes based on client IP ranges (CIDR blocks) — used for ISP-specific or enterprise-specific routing optimization

### How do you monitor private resources with Route 53 health checks? #card #aws #dns
Route 53 health checkers are outside the VPC, so they can't access private endpoints directly. Instead, create a CloudWatch metric + alarm for the private resource, then create a Route 53 health check that monitors that CloudWatch alarm.

Route 53 Calculated Health Checks :: Combine multiple health checks into one using AND, OR, or NOT logic — useful for monitoring complex architectures

A Public Hosted Zone resolves from the ==internet==, while a Private Hosted Zone resolves only within a ==VPC==.

Simple routing returns multiple values and the client chooses ==randomly== — no health checks are supported.

# AWS Messaging - SQS, SNS, Kinesis

Decoupling applications using messaging and streaming services.

## SQS - Simple Queue Service

Fully managed message queuing. Oldest AWS service. Used to **decouple** applications.

### Standard Queue

- Unlimited throughput, unlimited messages
- Default retention: 4 days (max 14 days)
- Low latency (<10ms)
- Max message size: **256KB**
- Can have **duplicate messages** (at-least-once delivery)
- Can have **out-of-order** messages (best-effort ordering)

### FIFO Queue

- **First In First Out** — ordering guaranteed
- Limited throughput: 300 msg/s without batching, 3,000 with
- **Exactly-once** send (deduplication)
- Queue name must end in `.fifo`

### Key Features

- **Visibility Timeout** — after consumer reads a message, it's invisible to others (default 30s). If not deleted in time, it becomes visible again (processed twice). Use `ChangeMessageVisibility` API.
- **Dead Letter Queue (DLQ)** — messages failing after `MaxReceiveCount` go to DLQ for debugging
- **Long Polling** — wait for messages instead of empty responses. Decreases API calls (1-20s).
- **Message Filtering** — SNS subscription filter to SQS
- Consumers: [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]], [[50-59-aws-saa/51.13-aws-serverless|Lambda]], on-premises servers
- Can scale consumers via [[50-59-aws-saa/51.06-aws-asg|ASG]] using CloudWatch `ApproximateNumberOfMessages` metric

### SQS + Lambda

Lambda polls SQS. Set up DLQ to avoid infinite retry loops. For FIFO, a stuck message can block the whole queue.

## SNS - Simple Notification Service

Pub/sub model — one message to many receivers.

- **Topic** — logical access point. Publishers send to topic.
- Up to **12.5 million subscriptions** per topic, 100,000 topics
- Subscribers: SQS, [[50-59-aws-saa/51.13-aws-serverless|Lambda]], Kinesis Firehose, email, SMS, HTTP endpoints
- **FIFO Topics** — ordering + deduplication (only SQS FIFO as subscriber)

### Fan Out Pattern

SDK → **SNS** → multiple SQS queues (instead of SDK → SQS directly). If app crashes sending to multiple SQS queues, some may miss the message.

### SNS + Lambda

Lambda retries asynchronously 3 times. Lambda can set up its own DLQ to SQS.

## Kinesis

Real-time streaming data collection, processing, and analysis.

### Kinesis Data Streams

- Real-time data streaming with **shards** (each 1MB/s in, 2MB/s out)
- Retention: 1-365 days
- Immutable data (can't delete once inserted)
- Producers: SDK, Kinesis Producer Library, Kinesis Agent
- Consumers: SDK, KCL, [[50-59-aws-saa/51.13-aws-serverless|Lambda]], Kinesis Firehose

### Kinesis Data Firehose

- Near real-time delivery (60s buffer or 1MB buffer)
- Fully managed, auto-scaling, serverless
- Destinations: [[50-59-aws-saa/51.08-aws-s3|S3]], [[50-59-aws-saa/51.09-aws-databases#Redshift|Redshift]], OpenSearch, 3rd-party, custom HTTP
- Can transform data using Lambda
- No replay capability

### Kinesis Data Analytics

- **SQL** (deprecated) — ingest from Firehose/Datastream, SQL queries, output to Firehose/Datastream
- **Apache Flink** (recommended) — Java, SQL, Scala on AWS. Complex queries. Ingests from Datastream or MSK.

### Kinesis vs SQS

| Feature            | Kinesis Streams                  | SQS                     |
| ------------------ | -------------------------------- | ----------------------- |
| **Consumer model** | Shared (pull) or enhanced (push) | Pull                    |
| **Replay**         | Yes (data persisted)             | No (deleted after read) |
| **Ordering**       | Per shard                        | FIFO only               |
| **Throughput**     | Provisioned (per shard)          | Unlimited               |

## Amazon MSK (Managed Kafka)

Alternative to Kinesis. Fully managed Kafka on AWS.

- Multi-AZ in your [[50-59-aws-saa/51.16-aws-vpc|VPC]], data on [[50-59-aws-saa/51.04-aws-ec2-storage|EBS]]
- **Data kept forever** (vs Kinesis max 1 year)
- Higher message size limit (1MB default, configurable to 10MB vs Kinesis 1MB)
- Can only **add** partitions (Kinesis can remove shards)
- Has serverless option
- Consumed by: Flink, EC2, Lambda, ECS, Glue

## Flashcards

What is the SQS message visibility timeout? :: Time a message is invisible after being polled (default 30s) — if not deleted, it reappears for other consumers

SQS is ==pull==-based; SNS is ==push==-based.

Kinesis Data Streams ::: Real-time data streaming with configurable shard-based throughput (1 MB/s in, 2 MB/s out per shard) and 1-365 day retention — consumers manage their own reading position

What is the SNS + SQS Fan Out pattern? :: Push a message to SNS once and have it delivered to multiple SQS queues subscribed to the topic — fully decoupled, no data loss

### SQS vs SNS vs Kinesis — when to use each? #card #aws #messaging

SQS: decouple producers/consumers, queue-based (pull). SNS: pub/sub fan-out to multiple subscribers (push). Kinesis: real-time streaming with ordering, replay, and multiple consumers reading the same stream.

SQS max message size is ==256 KB==. For larger messages, use the SQS Extended Client Library to store payload in S3.

SQS message retention is ==4 days== by default, configurable up to ==14 days==.

SQS FIFO queues guarantee ==exactly-once== processing and ==ordering== — throughput limited to 300 msg/s (or 3000 with batching).

SQS Dead Letter Queue (DLQ) :: After a message exceeds the MaxReceiveCount (failed processing attempts), it's moved to the DLQ for debugging — set DLQ retention to 14 days

SQS Long Polling :: Waits up to 20 seconds for messages to arrive before returning empty — reduces API calls and latency vs short polling

SNS subscribers can be :: SQS queues, Lambda functions, Kinesis Data Firehose, HTTP/HTTPS endpoints, email, SMS, and mobile push

SNS FIFO topics can only have ==SQS FIFO queues== as subscribers.

Kinesis Data Firehose :: Near-real-time delivery service (60 second buffer) to S3, Redshift, Elasticsearch, and HTTP endpoints — fully managed, auto-scaling, no data storage

### Kinesis Data Streams vs Kinesis Data Firehose — when to use each? #card #aws #messaging

Data Streams: real-time (200ms), custom consumer code, 1-365 day retention, replay capability. Firehose: near-real-time (60s buffer), fully managed delivery to S3/Redshift/ES, no data storage, auto-scaling, no replay.

Amazon MQ :: Managed message broker for Apache ActiveMQ and RabbitMQ — use when migrating from on-premises and can't rewrite to SQS/SNS

Amazon MQ runs on servers with ==Multi-AZ failover== and stores data on EBS. Unlike SQS/SNS, it has a ==finite message size limit of 1 MB== (ActiveMQ default).

Kinesis Data Streams retention is ==1 day== by default, configurable up to ==365 days==.

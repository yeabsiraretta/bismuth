# AWS Monitoring, Audit & Performance

Monitoring, logging, and audit services in [[50-59-aws-saa/51.00-aws-saa-overview|AWS]].

## CloudWatch Metrics

Collect and track metrics for every AWS service. Metric = variable to monitor.

- Metrics belong to **namespaces**
- **Dimensions** — attributes (instance ID, environment, etc.), max 30 per metric
- Default monitoring: **5 minutes**; Detailed monitoring: **1 minute** (paid, more data points for [[50-59-aws-saa/51.06-aws-asg|ASG]] scaling)
- [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] memory/disk usage are **not** default metrics — need a custom metric via Unified Agent
- Custom metrics: `PutMetricData` API, resolution from 1 second to 1 minute

### CloudWatch Alarms

- States: OK, INSUFFICIENT_DATA, ALARM
- Targets: [[50-59-aws-saa/51.06-aws-asg|ASG]] actions, [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] actions (stop/terminate/reboot), [[50-59-aws-saa/51.12-aws-messaging|SNS]] notifications
- **Composite Alarms** — combine multiple alarms with AND/OR conditions (reduce noise)
- Can be created via `set-alarm-state` CLI for testing

## CloudWatch Logs

Centralized log collection and analysis.

- **Log Groups** — usually per application
- **Log Streams** — per instance/container/log file
- Define expiration (never expire to 10 years)
- Export to [[50-59-aws-saa/51.08-aws-s3|S3]] (batch, up to 12 hours), or real-time via Logs Subscriptions → Lambda, [[50-59-aws-saa/51.12-aws-messaging|Kinesis]]
- Cross-account aggregation via Subscription Filters

### Log Sources

SDK, [[50-59-aws-saa/51.14-aws-containers|ECS]], [[50-59-aws-saa/51.13-aws-serverless|Lambda]], [[50-59-aws-saa/51.16-aws-vpc#VPC Flow Logs|VPC Flow Logs]], [[50-59-aws-saa/51.10-aws-route53|Route 53]] DNS logs, CloudTrail, API Gateway, Elastic Beanstalk.

### CloudWatch Logs Insights

Fully managed query service for CloudWatch Logs. Dashboard-ready. Purpose-built query language.

### CloudWatch Agent

Install on [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] or on-prem for additional metrics:
- **Unified Agent** (newer, recommended) — CPU, disk, RAM, net, processes, swap. Sends logs to CloudWatch. Configurable via SSM Parameter Store.

## EventBridge (CloudWatch Events)

Serverless event bus for reacting to AWS events or schedules.

- **Sources**: AWS services, custom apps, SaaS partners (Zendesk, Datadog, etc.)
- **Targets**: [[50-59-aws-saa/51.13-aws-serverless|Lambda]], [[50-59-aws-saa/51.12-aws-messaging|SQS/SNS/Kinesis]], Step Functions, CodePipeline, [[50-59-aws-saa/51.14-aws-containers|ECS]] tasks
- **Schema Registry** — model event schema, generate code bindings
- **Archive & Replay** — replay archived events for debugging
- Resource-based policies — allow events from other accounts/regions
- Event bus: default (AWS), partner, or custom

## CloudTrail

Track **API calls** made by users, services, and CLI/SDK. Governance, compliance, audit.

- Enabled by default
- Event history: last **90 days** (send to S3/CloudWatch for longer retention)
- **Management Events** — operations on resources (on by default)
- **Data Events** — high-volume operations like [[50-59-aws-saa/51.08-aws-s3|S3]] GetObject, [[50-59-aws-saa/51.13-aws-serverless|Lambda]] Invoke (off by default)
- **Insights** — detect unusual activity automatically

## AWS Config

Audit and record **compliance** of resource configurations over time.

- Per-region service (can aggregate across regions/accounts)
- Store in [[50-59-aws-saa/51.08-aws-s3|S3]], query with [[50-59-aws-saa/51.18-aws-data-analytics#Athena|Athena]]
- **Config Rules** — managed (75+) or custom (Lambda). Evaluate on config change or periodically.
- Does NOT prevent actions — only evaluates compliance after the fact
- **Remediations** — auto-remediate non-compliant resources via SSM Automation Documents

### Config vs CloudTrail

- **CloudTrail** — "who made this API call?"
- **Config** — "is this resource compliant? how did it change over time?"

## Flashcards

What is the difference between CloudTrail and AWS Config? :: CloudTrail answers "who made this API call?" (audit trail); Config answers "is this resource compliant and how did it change over time?" (compliance)

CloudWatch custom metrics require a ==PutMetricData== API call with a minimum resolution of ==1 second== (high resolution).

Amazon EventBridge ::: Serverless event bus that routes events from AWS services, SaaS apps, and custom sources to targets using rules with filtering — successor to CloudWatch Events

CloudWatch Logs Insights lets you query logs with a ==purpose-built query language== — automatically discovers fields from JSON logs.

### CloudWatch vs CloudTrail vs Config — what does each do? #card #aws #monitoring
CloudWatch: metrics, logs, alarms, dashboards (performance monitoring). CloudTrail: API call audit trail — who did what, when (governance). Config: resource configuration history and compliance rules (drift detection).

CloudWatch default metric period is ==5 minutes== (free). Detailed monitoring is every ==1 minute== (paid).

CloudWatch Alarms can trigger :: EC2 actions (stop, terminate, reboot, recover), ASG scaling actions, or SNS notifications

CloudWatch Unified Agent :: Runs on EC2 and on-prem servers to collect both system-level metrics (RAM, disk, processes, network) and logs — more granular than default EC2 metrics

CloudTrail is enabled ==by default== and records API calls for the last ==90 days==. For longer retention, configure a trail to deliver to S3.

CloudTrail management events are logged ==by default==; data events (S3 object-level, Lambda invocations) must be ==explicitly enabled==.

AWS Config rules can be either ==AWS managed== (pre-built by AWS) or ==custom== (defined via Lambda). Config does NOT prevent actions — it only evaluates compliance.

Config Remediations :: Automatically fix non-compliant resources using SSM Automation documents — e.g., auto-enable encryption on unencrypted S3 buckets

AWS X-Ray :: Distributed tracing service for debugging and analyzing microservices — provides visual service maps, identifies bottlenecks, traces requests across services

CloudWatch Composite Alarms :: Combine multiple alarms using AND/OR conditions — reduces alarm noise by only triggering when multiple conditions are met

CloudWatch Contributor Insights :: Analyze log data to find the top-N contributors — useful for identifying heavy hitters like top IP addresses or URLs

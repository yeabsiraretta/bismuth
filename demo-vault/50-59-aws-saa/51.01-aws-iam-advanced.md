# AWS IAM - Advanced

Advanced [[50-59-aws-saa/51.01-aws-iam|IAM]] concepts including Organizations, SCPs, Identity Center, and Directory Services.

## AWS Organizations

Group many accounts under one organization with one main account.

- Member accounts can only belong to **one organization**
- Share billing, reserved instances, get volume discounts
- Organized into **Organizational Units (OUs)** — can nest OUs within OUs
- Multi-account > one account with multi-[[50-59-aws-saa/51.16-aws-vpc|VPC]]
- Use tagging standards for billing
- Enable [[50-59-aws-saa/51.19-aws-monitoring|CloudTrail]] on all accounts, send logs to central account
- Establish cross-account roles for admins

### Service Control Policies (SCP)

- [[50-59-aws-saa/51.01-aws-iam|IAM]] policies applied to OU or accounts to **restrict** permissions
- Do NOT apply to management account
- Everything disabled by default — must add explicit allow
- **Any deny overrides all allows**
- SCPs are inherited from parent OUs

## Advanced IAM Policy Conditions

- `aws:SourceIp` — restrict client IP for API calls
- `aws:RequestedRegion` — restrict API call regions
- `ec2:ResourceTag` — restrict based on tags
- `aws:MultiFactorAuthPresent` — force MFA
- `aws:PrincipalOrgID` — restrict to organization accounts

## IAM for S3

- `s3:ListBucket` — bucket-level: `"arn:aws:s3:::test"`
- `s3:PutObject`, `s3:DeleteObject` — object-level: `"arn:aws:s3:::test/*"`

## IAM Roles vs Resource-Based Policies

For cross-account API calls:

- **Resource-based policy** — you keep your current permissions
- **Assume a role** — you give up all permissions not in that role

Example: User A scans DynamoDB in Account A, writes to [[50-59-aws-saa/51.08-aws-s3|S3]] bucket in Account B → use resource-based policy on S3 so User A keeps DynamoDB permissions.

### EventBridge Permissions

- **SNS, SQS, Lambda, CloudWatch Logs, API Gateway** → resource-based policy
- **Kinesis, ECS task, Systems Manager** → IAM role

## IAM Permission Boundaries

Managed policy setting **maximum permissions** for an entity. Supported for users and roles, NOT groups.

Use cases:

- Delegate responsibilities to non-admins
- Let developers self-assign policies without privilege escalation
- Restrict a specific user instead of using Organizations/SCP

## IAM Identity Center (SSO)

SSO for all AWS accounts, organizations, business cloud apps (Office 365), SAML2.0 apps, and [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] Windows instances.

- Identity from built-in store or 3rd party (Active Directory)
- **Permission Sets** — IAM policy collections granting access across multiple accounts
- **ABAC** (Attribute Based Access Control) — permissions based on user attributes

## AWS Directory Services

- **AWS Managed Microsoft AD** — create AD in AWS, "trust" with on-prem AD, integrates with Identity Center out-of-the-box
- **AD Connector** — proxy to on-prem AD, users managed on-prem, supports MFA
- **Simple AD** — AD-compatible on AWS, cannot connect to on-prem, useful for Windows [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]]

Connecting Identity Center to self-managed AD:

- Use Managed Microsoft AD with two-way trust, **or**
- AD Connector (slightly more latency)

## AWS Control Tower

Set up and secure a compliant multi-account AWS environment using Organizations.

- **Preventive guardrails** — uses SCP (e.g., restrict regions)
- **Detective guardrails** — uses [[50-59-aws-saa/51.19-aws-monitoring|AWS Config]] (e.g., identify untagged resources)
- Monitor compliance via interactive dashboard

## Flashcards

What are Service Control Policies (SCPs)? :: IAM policies applied to OUs or accounts in AWS Organizations to restrict permissions — any deny overrides all allows

SCPs do ==NOT== apply to the management account of an AWS Organization.

For cross-account access, a ==resource-based policy== lets you keep your current permissions, while ==assuming a role== means you give up all permissions not in that role.

IAM Permission Boundaries ::: Managed policies that set the maximum permissions for a user or role — used to delegate responsibilities without privilege escalation

### How do SCP, Permission Boundary, and Identity Policy interact? #card #aws #iam

The effective permission is the intersection of all three: SCP (org-level max), Permission Boundary (account-level max), and Identity Policy (actual grants). A user must have permission in ALL three to perform an action.

AWS Organizations ::: A service to group many AWS accounts together under one main account for consolidated billing, reserved instance sharing, and volume discounts

A member account can only be part of ==one== AWS Organization at a time.

SCPs are ==inherited== from parent OUs to child OUs and accounts.

In Organizations, everything is ==disabled by default== in SCPs — you must add explicit allow statements.

What does the `aws:SourceIp` IAM condition key do? :: Restricts the client IP address from which API calls can be made

What does the `aws:RequestedRegion` IAM condition key do? :: Restricts which AWS regions API calls can be made to

What does `aws:MultiFactorAuthPresent` enforce? :: Forces MFA to be present before allowing the action described in the policy

`s3:ListBucket` is a ==bucket-level== permission using ARN `arn:aws:s3:::bucket`, while `s3:PutObject` is an ==object-level== permission using ARN `arn:aws:s3:::bucket/*`.

### Why use resource-based policy instead of assuming a role for cross-account S3 access? #card #aws #iam

When User A needs to scan DynamoDB in Account A AND write to S3 in Account B, use a resource-based policy on the S3 bucket. Assuming a role in Account B would cause User A to lose their DynamoDB permissions in Account A.

### Which EventBridge targets use resource-based policies vs IAM roles? #card #aws #iam

Resource-based policies: SNS, SQS, Lambda, CloudWatch Logs, API Gateway. IAM roles: Kinesis, ECS tasks, Systems Manager Run Command.

IAM Permission Boundaries are supported for ==users and roles==, but NOT for groups.

### What are the use cases for IAM Permission Boundaries? #card #aws #iam

1. Delegate responsibilities to non-admins (e.g., let them create IAM users). 2) Allow developers to self-assign policies without escalating privileges past a boundary. 3) Restrict one specific user without needing Organizations/SCP.

IAM Identity Center (SSO) :: A single sign-on service for all AWS accounts, organizations, business cloud apps (Office 365), SAML2.0 apps, and EC2 Windows instances

What are Permission Sets in IAM Identity Center? :: Collections of IAM policies that define which users can access resources across multiple AWS accounts — e.g., developers access dev but not prod

ABAC (Attribute Based Access Control) :: Fine-grained permissions based on user attributes stored in IAM Identity Center Identity Store — set permissions once, then modify access by changing attributes

### What are the three AWS Directory Services options? #card #aws #iam

1. AWS Managed Microsoft AD — create AD in AWS with trust to on-prem AD, integrates with Identity Center. 2) AD Connector — proxy to on-prem AD, users managed on-prem, slightly more latency. 3) Simple AD — standalone AD on AWS, cannot connect to on-prem, useful for Windows EC2.

AWS Control Tower ::: A service to set up and secure a compliant multi-account AWS environment using Organizations, with preventive guardrails (SCP) and detective guardrails (AWS Config)

Preventive guardrails in Control Tower use ==SCPs==, while detective guardrails use ==AWS Config==.

What does `aws:PrincipalOrgID` condition key do? :: Restricts access to only accounts that are members of a specific AWS Organization

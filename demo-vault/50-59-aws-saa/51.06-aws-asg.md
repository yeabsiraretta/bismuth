# AWS Auto Scaling Groups

Automatically scale [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] instances to match load, integrated with [[50-59-aws-saa/51.05-aws-elb|ELB]].

## What is Auto Scaling?

- **Scale out** — add instances to match increased load
- **Scale in** — remove instances to match decreased load
- Ensures minimum and maximum number of machines running
- Automatically register/deregister instances to [[50-59-aws-saa/51.05-aws-elb|load balancers]]
- Free service (pay only for underlying EC2 instances)

## Launch Configuration / Launch Template

Define: AMI, instance type, [[50-59-aws-saa/51.02-aws-ec2-fundamentals#Security Groups|SG]], SSH key pair, [[50-59-aws-saa/51.01-aws-iam|IAM role]], user data, [[50-59-aws-saa/51.04-aws-ec2-storage|EBS]] volumes, network/subnet info, [[50-59-aws-saa/51.05-aws-elb|LB]] info.

- **Launch Templates** (newer) — versioned, supports T2 unlimited, [[50-59-aws-saa/51.02-aws-ec2-pricing#Spot Instances|Spot]] fleets, recommended
- **Launch Configurations** (legacy) — must recreate each time

## ASG Parameters

- Min size, Max size, Desired capacity, Cooldown period (default 300s)
- Health checks: EC2 (default) or ELB (recommended with LB)
- If instance is unhealthy, ASG terminates and launches a replacement

## Scaling Policies

### Dynamic Scaling

- **Target Tracking** — set a target metric (e.g., avg CPU at 40%), ASG adjusts automatically
- **Simple Scaling** — add/remove N instances when CloudWatch alarm triggers (e.g., CPU > 70% → add 2)
- **Step Scaling** — add/remove different amounts based on alarm severity ranges

### Scheduled Scaling

Set min capacity at known peak times (e.g., 10 AM on Fridays).

### Predictive Scaling

ML-based forecasting using historical load patterns.

## Scaling Cooldown

After a scaling activity, ASG enters a **cooldown period** (default 300 seconds) during which it won't launch/terminate additional instances. Use a ready-to-use AMI to reduce configuration time.

## Termination Policy

Default: find AZ with most instances → delete instance with oldest launch config. ASG tries to **balance across AZs** first.

## Lifecycle Hooks

Perform custom actions before an instance goes into service (Pending state) or before termination (Terminating state).

## ASG + ELB Integration

ASG can automatically register new instances with [[50-59-aws-saa/51.05-aws-elb|ELB]] target groups and use ELB health checks to determine instance health.

## Flashcards

Auto Scaling Group (ASG) :: Automatically scales EC2 instances in/out to match load — integrates with ELB for automatic registration and health checks

ASG scaling is free — you only pay for the ==underlying EC2 instances== launched.

Launch Templates are ==versioned== and recommended over Launch Configurations, which are ==legacy== and must be recreated each time.

What are the three types of dynamic scaling policies? :: Target Tracking (maintain a target metric), Simple Scaling (add/remove fixed count on alarm), Step Scaling (variable count based on alarm severity)

Target Tracking Scaling :: Set a target metric value (e.g., keep average CPU at 40%) and ASG automatically adjusts instance count to maintain it

The default ASG scaling cooldown period is ==300 seconds==.

Scheduled Scaling :: Increase min capacity at known peak times — e.g., set min to 10 instances at 10 AM on Fridays when you expect heavy traffic

Predictive Scaling :: Uses ML to forecast load based on historical patterns and proactively schedules the right number of instances

### What is the ASG default termination policy? #card #aws #asg

1. Find the AZ with the most instances. 2) Delete the instance with the oldest launch configuration or template. The goal is to keep instances balanced across AZs first.

ASG health checks default to ==EC2 status checks==, but should be set to ==ELB health checks== when used with a load balancer.

ASG Lifecycle Hooks ::: Custom actions performed before an instance enters service (Pending state) or before termination (Terminating state) — e.g., run setup scripts or drain connections

### What good scaling metrics can ASG use? #card #aws #asg

1. CPU Utilization — average across instances. 2) RequestCountPerTarget — keep requests per instance stable. 3) Average Network In/Out — for network-bound apps. 4) Custom CloudWatch metrics.

If an instance is determined ==unhealthy==, ASG automatically terminates it and launches a replacement.

# AWS EC2 - Instance Storage

Storage options for [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] instances.

## EBS - Elastic Block Storage

Persistent block storage — a **network drive** (not physical). Uses network to communicate, so some latency.

- Bound to a **specific AZ** — to move, snapshot first
- Can only mount to **one instance** at a time (CCP level)
- Replicated within AZ for protection
- Can modify volume type on the fly (may need repartition)
- Additional (non-root) EBS can be detached without stopping instance

### EBS Defaults

- Root EBS volumes **deleted** on termination; additional volumes are **not**
- Unattached EBS status = "available"

### EBS Snapshots

- Point-in-time backup of an EBS volume (stored on [[50-59-aws-saa/51.08-aws-s3|S3]])
- **Incremental** — only changed blocks since last snapshot
- Recommended to detach before snapshotting
- **Archive** — 75% cheaper, 24-72 hours to restore
- **Recycle bin** — retain deleted snapshots (1 day to 1 year)
- **Fast Snapshot Restore** — no latency on first use (expensive)
- Can't delete snapshot of EBS used as root device of registered AMI

### Change Volume AZ

Create snapshot → Turn into AMI → Launch in different AZ. Can also copy AMIs across regions.

### EBS Volume Types

| Type        | Category             | Use Case                    | Boot? |
| ----------- | -------------------- | --------------------------- | ----- |
| **gp2/gp3** | General SSD          | Boot, dev/test, desktops    | Yes   |
| **io1/io2** | Provisioned IOPS SSD | Databases, >16K IOPS        | Yes   |
| **st1**     | Throughput HDD       | Data warehouse, big data    | No    |
| **sc1**     | Cold HDD             | Infrequent access, cheapest | No    |

- **gp3**: size and throughput independent; **gp2**: size and IOPS linked
- **io2**: more durability, more IOPS/GiB at same price as io1
- io1/io2 use [[50-59-aws-saa/51.03-aws-ec2-advanced#EC2 Nitro|Nitro]] for >32K IOPS

### EBS Multi-Attach (io1/io2 only)

Attach same EBS to **multiple EC2 in same AZ** (up to 16). Must use cluster-aware filesystem.

### EBS Encryption

- Data at rest and in-flight encrypted, snapshots auto-encrypted
- Minimal latency impact
- To encrypt existing: snapshot → copy with encryption → create new volume
- Can only share **unencrypted** snapshots (publicly or with specific accounts)

### Delete on Termination

Root volume deleted by default; other attached volumes not. Disable to preserve root on termination.

## Amazon Machine Image (AMI)

Virtual machine image containing bootable OS. Types: **Public** (AWS), **Your own**, **Marketplace**.

- Built for a **specific region** (copy across regions via snapshot)
- AMI Process: Start EC2 → Customize → Stop → Build AMI → Launch from AMI
- Storage types: **Instance Store** (template in S3) or **EBS-backed** (from EBS snapshot)
- AWS doesn't copy launch permissions, tags, or S3 permissions when copying AMIs

## EC2 Instance Store

High-performance **ephemeral** hardware disk (better I/O than EBS).

- Data lost on: disk failure, instance stop, instance termination
- Rebooting does **NOT** cause data loss
- Good for: buffer, cache, scratch data, replicated data
- For >256K IOPS, use Instance Store
- Can only attach at launch, not after

## EFS - Elastic File System

Linux-based **NFS** mountable across multiple EC2s in **different AZs**.

- Highly available, scalable, expensive, pay per use
- Supports NFS v4; requires SG for access
- Encryption at rest with KMS; elastic storage (grows/shrinks automatically)
- Use for: content management, web serving, data sharing, WordPress

### EFS Performance

- **General Purpose** (default) — low latency (web servers)
- **Max I/O** — higher throughput, more concurrent ops (big data)
- Storage Tiers: Standard (frequent) → Infrequent Access (cheaper storage, retrieval cost)

## EBS vs EFS Comparison

| Feature         | EBS                        | EFS                          | Instance Store   |
| --------------- | -------------------------- | ---------------------------- | ---------------- |
| **Scope**       | Single instance, single AZ | Multiple instances, multi-AZ | Single instance  |
| **Persistence** | Persistent                 | Persistent                   | Ephemeral        |
| **OS Support**  | Any                        | Linux only                   | Any              |
| **Planning**    | Provision capacity         | Pay per use                  | Fixed at launch  |
| **Performance** | Up to 64K IOPS             | 10GB+/s throughput           | Millions of IOPS |

## Flashcards

EBS is bound to a ==specific AZ== — to move it, you must snapshot first.

What is the key difference between EBS and Instance Store? :: EBS is persistent network storage; Instance Store is ephemeral high-performance hardware disk that loses data on stop/terminate

Which EBS volume types can be used as boot volumes? :: Only gp2/gp3 (General SSD) and io1/io2 (Provisioned IOPS SSD)

EFS ::: Linux-based NFS file system mountable across multiple EC2 instances in different AZs — highly available, scalable, pay-per-use with elastic storage

EBS Multi-Attach is only available for ==io1/io2== volumes and limited to instances in the ==same AZ==.

### EBS vs EFS vs Instance Store — when to use each? #card #aws #storage

EBS: persistent block storage for single-instance workloads (locked to AZ). EFS: shared NFS for multi-instance access across AZs. Instance Store: ephemeral, highest IOPS for caches/buffers (data lost on stop).

EBS snapshots are ==incremental== — only blocks changed since the last snapshot are moved to S3.

You ==cannot== delete a snapshot of an EBS volume used as the root device of a registered AMI. You must deregister the AMI first.

EBS Snapshot Archive :: Moves snapshot to archive tier at 75% cheaper cost, but takes 24-72 hours to restore

Fast Snapshot Restore :: Forces full initialization of an EBS snapshot so there is no latency on first use — very expensive option

By default, root EBS volumes are ==deleted== on EC2 termination, but additional attached volumes are ==not deleted==.

gp3 volumes have ==independent== size and throughput settings, while gp2 volumes have size and IOPS ==linked== together.

io2 volumes offer more ==durability== and more IOPS per GiB at the ==same price== as io1.

Can you detach an additional (non-root) EBS volume without stopping the instance? :: Yes — only additional disks can be detached while the instance is running

### How do you move an EBS volume to a different AZ? #card #aws #storage

1. Create a snapshot of the volume. 2) Use the snapshot to create an AMI. 3) Launch a new EC2 instance from the AMI in the target AZ. You can also copy AMIs across regions.

EBS encryption has ==minimal== impact on latency. All data at rest and in-flight between instance and volume is encrypted.

To encrypt an unencrypted EBS volume :: Create a snapshot → Copy the snapshot with encryption enabled → Create a new volume or AMI from the encrypted snapshot

You can only share ==unencrypted== snapshots — either publicly or with specific AWS accounts.

AMIs are built for a ==specific region== but can be copied across regions via snapshot.

When copying an AMI, AWS does NOT copy ==launch permissions==, ==user-defined tags==, or ==S3 bucket permissions==.

Instance Store volumes can only be attached ==at launch==, not after the instance is running.

Rebooting an EC2 instance does ==NOT== cause Instance Store data loss — only stopping, terminating, or disk failure does.

For an IOPs requirement above ==256,000==, you must use EC2 Instance Store instead of EBS.

EFS supports ==NFS v4== protocol and requires a ==Security Group== to control access.

EFS storage is ==elastic== — it grows and shrinks automatically with no need to pre-provision capacity.

EFS General Purpose mode :: Default performance mode for latency-sensitive use cases like web servers and small files

EFS Max I/O mode :: Higher latency and throughput with more concurrent operations — used for big data and media processing

### What are the EFS storage tiers? #card #aws #storage

Standard (for frequently accessed files) and Infrequent Access (lower storage price but cost to retrieve). Use Lifecycle Management to automatically move files after N days.

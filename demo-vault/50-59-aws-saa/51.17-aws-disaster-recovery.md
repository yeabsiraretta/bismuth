# AWS Disaster Recovery & Migrations

Strategies for business continuity and moving to [[50-59-aws-saa/51.00-aws-saa-overview|AWS]].

## Key Concepts

- **RPO** (Recovery Point Objective) — how much data loss is acceptable (time between last backup and disaster)
- **RTO** (Recovery Time Objective) — how much downtime is acceptable (time between disaster and recovery)

## DR Strategies (cheapest → most expensive)

### 1. Backup & Restore

- High RPO/RTO, low cost
- Snapshot [[50-59-aws-saa/51.04-aws-ec2-storage|EBS]], [[50-59-aws-saa/51.09-aws-databases|RDS]], Redshift
- Use [[50-59-aws-saa/51.08-aws-s3|S3]] with lifecycle to Glacier for cheap storage
- Use Snowball for large data migrations
- Restore from backups when disaster hits

### 2. Pilot Light

- Small version of critical app always running (e.g., [[50-59-aws-saa/51.09-aws-databases|RDS]] replication)
- Scale up when disaster happens
- Faster than backup/restore

### 3. Warm Standby

- Full system running at minimum size
- Scale to production load during disaster
- More expensive than pilot light but faster RTO

### 4. Hot Site / Multi-Site (Active-Active)

- Full production scale in another region
- Lowest RTO (near-zero), most expensive
- Use [[50-59-aws-saa/51.10-aws-route53|Route 53]] for DNS failover

## Database Migration Service (DMS)

Migrate databases to AWS with minimal downtime. Source DB remains available.

- Must run on [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] instance
- **Homogeneous** (same engine) — direct migration
- **Heterogeneous** (different engine) — use **SCT** (Schema Conversion Tool) first
- **Continuous replication** via CDC (Change Data Capture)
- Sources: on-prem DBs, EC2 DBs, [[50-59-aws-saa/51.09-aws-databases|RDS]], [[50-59-aws-saa/51.08-aws-s3|S3]], DocumentDB, MongoDB, etc.
- Targets: same list + Kinesis Data Streams, [[50-59-aws-saa/51.18-aws-data-analytics|OpenSearch]], [[50-59-aws-saa/51.09-aws-databases|DynamoDB]], Neptune, Redshift

## Migration Services

### AWS Application Discovery Service

Gather info about on-prem servers for migration planning.

- **Agentless** — VM inventory, configuration, performance
- **Agent-based** — system + network activity data

### Application Migration Service (MGN)

Lift-and-shift for physical/virtual/cloud servers → AWS. Continuous replication.

### Snow Family

Physical devices for edge computing and data migration.

| Device                    | Storage              | Compute       | Use Case                  |
| ------------------------- | -------------------- | ------------- | ------------------------- |
| **Snowcone**              | 8 TB HDD / 14 TB SSD | 2 CPUs, 4GB   | Small, harsh environments |
| **Snowball Edge Storage** | 80 TB                | 40 vCPUs      | Large data transport      |
| **Snowball Edge Compute** | 42 TB                | 52 vCPUs, GPU | Edge computing            |
| **Snowmobile**            | 100 PB               | N/A           | Exabyte-scale transfer    |

- Can't import directly to Glacier — import to S3 first, lifecycle to Glacier
- OpsHub: GUI to manage Snow devices

### AWS DataSync

Move large amounts of data between on-prem and AWS, or AWS-to-AWS. Agent-based, incremental. Destinations: [[50-59-aws-saa/51.08-aws-s3|S3]], [[50-59-aws-saa/51.04-aws-ec2-storage|EFS]], FSx.

### AWS Transfer Family

Managed SFTP/FTPS/FTP service for [[50-59-aws-saa/51.08-aws-s3|S3]] or EFS. Multi-AZ. Integrates with existing auth systems.

## AWS Backup

Centrally manage backups across AWS services. Cross-region, cross-account with [[50-59-aws-saa/51.01-aws-iam-advanced#AWS Organizations|Organizations]].

- Supports: EC2, EBS, S3, RDS, Aurora, DynamoDB, DocumentDB, Neptune, EFS, FSx, Storage Gateway
- **Vault Lock** — WORM (Write Once Read Once) policy for compliance. Even root can't delete.

## Flashcards

What are the four DR strategies from cheapest to fastest? :: Backup & Restore (hours) → Pilot Light (minutes) → Warm Standby (minutes, smaller scale) → Multi-Site/Hot Site (real-time, full scale)

==RPO== is how much data you can afford to lose; ==RTO== is how long until the system is back online.

AWS DMS (Database Migration Service) ::: Continuously replicates data from source to target database — supports homogeneous and heterogeneous migrations, with Schema Conversion Tool (SCT) for different engine conversions

AWS Backup Vault Lock enforces a ==WORM== (Write Once Read Many) policy — even the root account cannot delete backups.

### What are the four DR strategies in order of cost and speed? #card #aws #disaster-recovery

Backup & Restore (cheapest, slowest RTO). Pilot Light (core services always running, scale up on disaster). Warm Standby (scaled-down full system). Multi-Site/Hot Site (full production in 2+ regions, near-zero RTO, most expensive).

Pilot Light :: Keep core services (like databases) always running in DR region — on disaster, quickly scale up the remaining services

Warm Standby :: Run a full but scaled-down version of your production system in DR region — faster RTO than Pilot Light but more expensive

Snowball Edge Storage Optimized :: 80 TB storage, 40 vCPUs — for large-scale physical data transport to AWS

Snowcone :: 8 TB HDD or 14 TB SSD — smallest Snow device for harsh, space-constrained environments

Snowmobile :: 100 PB capacity — for exabyte-scale data transfer, literally a shipping container truck

Snow Family devices ==cannot== import data directly to Glacier — import to S3 first, then use lifecycle policy to move to Glacier.

AWS DataSync :: Moves large amounts of data between on-premises and AWS or AWS-to-AWS — agent-based, incremental transfers to S3, EFS, or FSx

AWS Transfer Family :: Managed FTP/SFTP/FTPS service for transfers into S3 or EFS — Multi-AZ, integrates with existing authentication systems

### When do you use Snow Family vs DataSync vs Transfer Family? #card #aws #migration

Snow Family: offline physical data migration (TB to PB scale, no network). DataSync: online bulk data migration with agent (incremental, scheduled). Transfer Family: managed FTP/SFTP for users who need familiar file transfer protocols.

DMS with SCT (Schema Conversion Tool) is required for ==heterogeneous== migrations (different DB engines). For ==homogeneous== migrations (same engine), SCT is not needed.

AWS Backup supports :: EC2, EBS, S3, RDS, Aurora, DynamoDB, DocumentDB, Neptune, EFS, FSx, and Storage Gateway

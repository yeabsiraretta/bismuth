# AWS IAM - Identity & Access Management

Manage users and their level of access to the AWS console and APIs. IAM is a [[50-59-aws-saa/51.00-aws-saa-overview#Global Services|global service]].

## Key Terminology

- **Users** — end users (people, orgs). Global scope. Can belong to multiple groups.
- **Groups** — collection of users. Only contain users, NOT other groups. Members inherit group permissions.
- **Policies** — JSON documents defining what a User/Group/Role can do. Follow **least privilege principle**.
- **Roles** — assigned to AWS resources (an "account" for software/services).

## Root Account

- Created by default — "god mode"
- **Never** use for daily tasks; only for initial AWS account setup
- Root sign-in URL differs from IAM sign-in URL
- Key-pair generated is account-wide and managed by root

## Users & Access

- **Power User Access** — access to all AWS services except IAM group/user management
- New users can't log in until given username + password
- Secret key + ID only allows **programmatic access** (CLI/SDK)
- Users inherit policies from their groups

## IAM Policies Structure

```json
{
  "Version": "2012-10-17",
  "Id": "optional-policy-id",
  "Statement": [
    {
      "Sid": "optional-statement-id",
      "Effect": "Allow|Deny",
      "Principal": "account/user/role",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::bucket/*"],
      "Condition": {}
    }
  ]
}
```

- **Explicit deny** always overrides any conflicting allow
- **Version** — always `"2012-10-17"`
- **Condition** — optional: when the policy is in effect

## IAM Roles for Services

AWS services need permissions to act on your behalf. Assign an [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] instance a role instead of using `aws configure` with personal credentials.

## Multi-Factor Authentication (MFA)

- **Virtual MFA** — Google Authenticator, Authy
- **U2F Security Key** — YubiKey (one key supports multiple root + IAM users)
- Use password + security device combination

## Security Best Practices

- Strong password policy + enforce rotation
- Use and enforce MFA
- One AWS user = one physical person
- Assign users to groups, assign permissions to groups
- Create and use roles for AWS services
- Use access keys for programmatic access only
- **Never** share IAM users or access keys
- Audit with IAM Credentials Report + IAM Access Advisor

## Security Tools

- **IAM Credentials Report** (account level) — lists all users and credential status
- **IAM Access Advisor** (user level) — shows granted permissions and last access time. Use to revise policies (least privilege).

## AWS CLI

- `aws configure` stores `ACCESS_KEY_ID` and `SECRET_ACCESS_KEY` in `~/.aws/` — **security risk**
- Use [[50-59-aws-saa/51.01-aws-iam#IAM Roles for Services|IAM roles]] instead — more secure, easier to manage
- Roles can be assigned before or after [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] creation

## Flashcards

IAM :: AWS service for managing users, groups, roles, and their access permissions

IAM Users ::: People or applications that need access to AWS resources

IAM is a ==global== service, not region-scoped.

New IAM users have ==no permissions== by default (least privilege).

The maximum object size in IAM policies is ==5120 characters== per managed policy.

What are the 4 key IAM components?
#card
Users, Groups, Policies, and Roles

IAM Groups can only contain ==users==, not other groups.

Power User Access :: Access to all AWS services EXCEPT management of groups and users within IAM

A new IAM user can't login to the console until given ==username and password==. The secret key and ID only allow programmatic access.

The root account sign-in URL is ==different from== the IAM account sign-in URL.

IAM Policy Version field :: Always set to `"2012-10-17"` — the policy language version

An explicit ==deny== in an IAM policy always overrides any conflicting allow.

### What are the components of an IAM Policy Statement? #card #aws #iam

Sid (optional identifier), Effect (Allow/Deny), Principal (account/user/role), Action (list of actions), Resource (list of resources), Condition (optional conditions for when policy is in effect).

### What MFA device options does AWS support? #card #aws #iam

1. Virtual MFA (Google Authenticator, Authy). 2) U2F Security Key (YubiKey — one key supports multiple root and IAM users).

Why should you never use `aws configure` on an EC2 instance? :: The ACCESS_KEY_ID and SECRET_ACCESS_KEY are stored in `~/.aws/` directory, allowing anyone with access to the instance to use your credentials. Use IAM roles instead.

IAM Credentials Report :: Account-level report listing all users and the status of their various credentials

IAM Access Advisor :: User-level tool showing service permissions granted to a user and when those permissions were last accessed — used to revise policies for least privilege

### What are the IAM security best practices? #card #aws #iam

1. Only use root for initial setup. 2) One AWS user = one physical person. 3) Assign users to groups. 4) Strong password policy + MFA. 5) Create roles for AWS services. 6) Use access keys only for CLI/SDK. 7) Audit with Credentials Report and Access Advisor. 8) Never share IAM users or access keys.

IAM Roles ::: Assigned to AWS resources to perform actions on your behalf — essentially an account for a piece of software, can be assigned before or after EC2 creation

# AWS Elastic Load Balancers

Servers that forward traffic to multiple [[50-59-aws-saa/51.02-aws-ec2-fundamentals|EC2]] instances downstream.

## Why Use a Load Balancer?

- Spread load across instances
- Single point of access (DNS)
- Handle downstream failures via **health checks** (port + route, 200 = OK)
- Separate public/private traffic
- SSL termination (HTTPS)
- Doesn't need to be internet-facing
- Need at least **2 public subnets** for internet-facing ELB

## Load Balancer Security Groups

Link SG of instances to SG of load balancer. Users connect from anywhere → traffic enters EC2 **only from the LB**.

Instances monitored as **InService** or **OutOfService**. If app stops responding, ELB returns **504 (Gateway Timeout)**.

## Types of ELB

### Application Load Balancer (ALB) — Layer 7

- HTTP, HTTPS, WebSocket
- **Static DNS** name (no static IP)
- Route by: path, hostname, query strings, HTTP headers
- Target groups: EC2 (via [[50-59-aws-saa/51.06-aws-asg|ASG]]), ECS tasks, Lambda, private IPs
- Health checks at target group level
- Supports redirects (HTTP → HTTPS)
- Use **X-Forwarded-For** header for client IP
- Use for: microservices, containers (Docker, ECS)

### Network Load Balancer (NLB) — Layer 4

- TCP/UDP, **ultra-low latency**, millions of req/sec
- **Static DNS + static IP** (one per AZ, supports Elastic IP)
- Target groups: EC2, IPs, ALB
- Health checks: TCP, HTTP, HTTPS
- Use for: extreme performance, TCP/UDP traffic

### Gateway Load Balancer (GLB) — Layer 3

- IP packets, uses **GENEVE** protocol on port 6081
- Transparent network gateway + load balancer
- Target groups: EC2, private IPs
- Use for: firewalls, intrusion detection, 3rd-party network appliances

### Classic Load Balancer (CLB) — Legacy

- HTTP/HTTPS + TCP, static DNS name
- Layer 7 features (X-Forwarded, sticky sessions) + strict Layer 4
- Use when keeping costs down

## Sticky Sessions (Session Affinity)

Bind user session to specific target (CLB, ALB).

- **Application cookies**: custom (don't use AWSALB/AWSALBAPP/AWSALBTG) or app cookie (AWSALBAPP)
- **Duration-based cookies**: AWSALB (ALB), AWSELB (CLB)
- Con: can cause load imbalance

## Cross-Zone Load Balancing

Distributes load evenly across ALL targets in multiple AZs (vs per-LB AZ distribution).

| Type | Default                         | Inter-AZ Cost |
| ---- | ------------------------------- | ------------- |
| ALB  | Always on (disable at TG level) | Free          |
| NLB  | Disabled                        | Paid          |
| CLB  | Disabled                        | Paid          |

## SSL/TLS Certificates

- Users → LB via HTTPS (encrypted); LB uses **X.509 certificate**
- **SNI** (Server Name Indication) — multiple SSL certs on one LB. ALB, NLB, CloudFront only.
- Manage with [[50-59-aws-saa/51.15-aws-security#AWS Certificate Manager|ACM]]

## Connection Draining / Deregistration Delay

Time for in-flight requests to complete while instance de-registers. New requests not sent. Set low for short requests.

## Path Patterns

Configure ALB listener to route by URL path.

## X-Forwarded-For Header

LB uses its own internal IP → use `X-Forwarded-For` to get client IP.

## Flashcards

What are the three modern ELB types and their OSI layers? :: ALB (Layer 7 — HTTP), NLB (Layer 4 — TCP/UDP), GLB (Layer 3 — IP/GENEVE)

ALB provides a ==static DNS== name but no static IP; NLB provides both ==static DNS== and ==static IP== (one per AZ).

SNI (Server Name Indication) ::: Technology that allows multiple SSL certificates on one load balancer by indicating the target hostname at TLS handshake — supported by ALB, NLB, and CloudFront only

Cross-Zone Load Balancing is ==always on== for ALB (free) but ==disabled by default== for NLB and CLB (paid).

### When do you choose ALB vs NLB vs GLB? #card #aws #networking

ALB: HTTP/HTTPS Layer 7 routing (path, host, header). NLB: TCP/UDP Layer 4 for extreme performance (millions of req/s, static IP). GLB: Layer 3 for transparent network appliances (firewalls, IDS/IPS).

An internet-facing ELB needs at least ==2 public subnets== to operate.

If an application stops responding behind an ELB, the LB returns a ==504 Gateway Timeout== error.

What header does an ALB use to pass the client's real IP to the EC2 instance? :: `X-Forwarded-For` header — because the LB forwards traffic using its own internal IP

ALB target groups can include :: EC2 instances (via ASG), ECS tasks, Lambda functions, and private IP addresses

NLB target groups can include :: EC2 instances, private IP addresses, and ALBs (for combining Layer 4 and Layer 7 capabilities)

Gateway Load Balancer uses the ==GENEVE== protocol on port ==6081==.

### What are the two types of sticky session cookies? #card #aws #elb

1. Application cookies: custom (user-defined name, can't use AWSALB/AWSALBAPP/AWSALBTG) or application-generated (AWSALBAPP). 2) Duration-based cookies: AWSALB (ALB) or AWSELB (CLB). Both bind a user to a specific target but can cause load imbalance.

Connection Draining (Deregistration Delay) :: Time allowed for in-flight requests to complete while an instance de-registers from the ELB — new requests are not routed to it; set low for short-lived requests

ELB health checks use a ==port== and a ==route== (e.g., `/health`), and a ==200 OK== response means healthy.

Instances behind an ELB are monitored as either ==InService== or ==OutOfService==.

### How do you secure EC2 instances behind a load balancer? #card #aws #elb

Configure the EC2 Security Group to only accept traffic from the Load Balancer's Security Group — not from the internet directly. Users connect to the LB, which forwards traffic to instances.

ALB can route based on ==path==, ==hostname==, ==query strings==, and ==HTTP headers==.

NLB supports ==Elastic IPs== — one per AZ — making it ideal when you need a known static IP for whitelisting.

### Why would you put an ALB behind an NLB? #card #aws #elb

To get NLB's static IP and Layer 4 performance while still using ALB's Layer 7 routing features. NLB can target ALB instances in its target groups.

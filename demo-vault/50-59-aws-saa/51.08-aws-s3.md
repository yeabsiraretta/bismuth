# AWS S3 - Simple Storage Service

Infinitely scaling object storage. One of the most important AWS services.

## Key Concepts

- **Buckets** — directories for objects. Globally unique name, defined at region level.
- **Objects** — files stored in buckets. Key = full path (prefix + object name).
- Max object size: **5TB**. Uploads >5GB must use **multi-part upload**.
- Objects have: metadata, tags (up to 10), version ID (if versioning enabled).

## S3 Security

### Bucket Policies

- JSON-based resource policies attached to buckets
- Grant public access, force encryption, cross-account access
- Use for: public bucket, force upload encryption, cross-account

### Access Control

- [[50-59-aws-saa/51.01-aws-iam|IAM]] policies attached to users/roles
- **Bucket ACLs** — less common, can be disabled
- **Object ACLs** — finer grain, can be disabled
- An IAM principal can access S3 if: IAM permissions ALLOW **or** resource policy ALLOWS, **and** there is no explicit DENY

### Block Public Access

Account-level setting to prevent public buckets. Enabled by default. Leave on unless you explicitly need public access.

## S3 Versioning

- Enabled at **bucket level**
- Same key overwrite increments version
- Best practice to version — protects against unintended deletes
- Deleting a versioned object adds a **delete marker** (reversible)
- Suspending versioning doesn't delete previous versions

## S3 Replication

- **CRR** (Cross-Region Replication) — compliance, lower latency, cross-account
- **SRR** (Same-Region Replication) — log aggregation, live replication between accounts
- Requires: versioning enabled on both, proper [[50-59-aws-saa/51.01-aws-iam|IAM]] permissions, can be cross-account
- Asynchronous replication; only new objects replicated after enabling
- **S3 Batch Replication** for existing objects
- No chaining: Bucket 1 → Bucket 2 → Bucket 3 doesn't auto-replicate 1→3
- With [[50-59-aws-saa/51.15-aws-security#KMS|KMS]]: enable SSE-KMS option, specify target key, need IAM role with kms:Decrypt/Encrypt

## S3 Storage Classes

| Class | Use Case | Retrieval | AZs |
|-------|----------|-----------|-----|
| **Standard** | Frequently accessed | Instant | 3+ |
| **Standard-IA** | Infrequent, rapid access | Instant | 3+ |
| **One Zone-IA** | Infrequent, non-critical | Instant | 1 |
| **Glacier Instant** | Archive, instant access | Instant | 3+ |
| **Glacier Flexible** | Archive, mins to hours | 1-12 hrs | 3+ |
| **Glacier Deep Archive** | Long-term archive | 12-48 hrs | 3+ |
| **Intelligent-Tiering** | Unknown/changing patterns | Auto-managed | 3+ |

### Lifecycle Rules

Automate transitions between classes and expiration of objects. Configure by prefix or tags.
- **Transition actions** — move between classes after N days
- **Expiration actions** — delete old versions, incomplete uploads

## S3 Performance

- **Baseline**: 3,500 PUT/COPY/POST/DELETE + 5,500 GET/HEAD per prefix per second
- **Multi-Part Upload** — recommended >100MB, required >5GB, parallelizes uploads
- **S3 Transfer Acceleration** — transfers to edge location, then AWS backbone to bucket
- **S3 Byte-Range Fetches** — parallelize downloads by requesting specific byte ranges

## S3 Encryption

- **SSE-S3** — AWS managed keys (AES-256). Default.
- **SSE-KMS** — [[50-59-aws-saa/51.15-aws-security#KMS|KMS]] managed keys. Audit trail. May hit KMS API rate limits.
- **SSE-C** — Customer-provided keys. Must use HTTPS. Keys sent in headers.
- **Client-Side Encryption** — encrypt before upload, decrypt after download.
- **Encryption in transit** — HTTPS endpoint. Required for SSE-C.

## S3 Access Points

Simplify managing access to shared datasets. Each access point has its own DNS, own policy.

## S3 Event Notifications

Trigger on: object created, removed, replicated, etc. Send to: SQS, SNS, Lambda, or [[50-59-aws-saa/51.19-aws-monitoring#EventBridge|EventBridge]].

## S3 Static Website Hosting

Host static websites directly from S3. Bucket policy must allow public reads. URL format: `<bucket>.s3-website-<region>.amazonaws.com`.

## S3 Pre-Signed URLs

Temporary access to objects. Inherits permissions of the user who generated it. Use for: premium content, temporary downloads/uploads.

## Cost Hierarchy

- Uploading to S3 is **free**
- ~$0.023/GB (cheapest storage option vs [[50-59-aws-saa/51.04-aws-ec2-storage|EBS]] ~$0.08/GB, EFS ~$0.08/GB)
- Transfer Acceleration: only pay for what gets accelerated

## Flashcards

S3 max object size :: 5TB (uploads >5GB must use multi-part upload)

S3 bucket names must be ==globally unique== across all AWS accounts.

S3 Versioning is enabled at the ==bucket== level.

What determines S3 object access when both IAM and bucket policies exist?
#card
IAM permissions ALLOW **or** resource policy ALLOWS, **and** no explicit DENY.

S3 Storage Classes (cheapest to most expensive) ::: Glacier Deep Archive, Glacier Flexible, Glacier Instant, S3-IA One Zone, S3-IA, S3 Intelligent-Tiering, S3 Standard

Uploading to S3 is =={free}== but retrieval and transfer incur costs.

S3 objects are identified by their ==key==, which is the full path (prefix + object name).

Multi-Part Upload is recommended for files >==100MB== and required for files >==5GB==.

Deleting a versioned S3 object adds a ==delete marker== which is reversible — it doesn't permanently delete previous versions.

Suspending S3 versioning does ==NOT== delete previous versions.

S3 CRR (Cross-Region Replication) ::: Replicate objects to a bucket in a different region — used for compliance, lower latency access, and cross-account replication

S3 SRR (Same-Region Replication) :: Replicate objects within the same region — used for log aggregation and live replication between production and test accounts

S3 replication only applies to ==new objects== after enabling — use ==S3 Batch Replication== for existing objects.

S3 replication does not support chaining: Bucket 1 → 2 → 3 does ==NOT== auto-replicate 1 → 3.

### What are the four S3 encryption methods? #card #aws #s3
1) SSE-S3: AWS managed keys (AES-256, default). 2) SSE-KMS: KMS managed, audit trail, may hit API limits. 3) SSE-C: customer-provided keys via HTTPS headers. 4) Client-Side: encrypt before upload.

SSE-C encryption requires using ==HTTPS== — keys are provided in request headers and never stored by AWS.

S3 Standard-IA vs One Zone-IA :: Standard-IA stores across 3+ AZs for durability; One Zone-IA stores in a single AZ — cheaper but data lost if AZ destroyed

Glacier Flexible Retrieval :: Archive storage with retrieval options: Expedited (1-5 min), Standard (3-5 hrs), Bulk (5-12 hrs)

Glacier Deep Archive :: Lowest cost storage for long-term archival — Standard retrieval takes 12 hours, Bulk takes 48 hours

S3 Intelligent-Tiering :: Automatically moves objects between tiers based on access patterns — small monthly monitoring fee, no retrieval charges

S3 Transfer Acceleration :: Uses edge locations and AWS backbone network to speed up transfers to S3 — pay only for data that gets accelerated

S3 Byte-Range Fetches :: Parallelize downloads by requesting specific byte ranges of an object — can also be used to retrieve just a partial file (e.g., first N bytes)

S3 baseline performance :: 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD requests per second per prefix

### What are S3 Lifecycle Transition and Expiration actions? #card #aws #s3
Transition actions move objects between storage classes after N days (e.g., Standard → IA after 60 days → Glacier after 180 days). Expiration actions delete old versions, delete markers, or incomplete multi-part uploads.

S3 Pre-Signed URLs :: Temporary URLs that inherit the permissions of the user who generated them — used for premium content, temporary downloads, or allowing specific users to upload

S3 Event Notifications can trigger :: SQS, SNS, Lambda, or EventBridge — on events like object creation, deletion, or replication

S3 static website URL format :: `<bucket-name>.s3-website-<region>.amazonaws.com` — bucket policy must allow public reads

S3 Access Points :: Simplify managing access to shared S3 datasets — each has its own DNS name and access policy, reducing complex bucket policy management

S3 Block Public Access is ==enabled by default== at the account level to prevent accidental public bucket exposure.
